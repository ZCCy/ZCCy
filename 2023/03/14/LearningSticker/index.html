Java

JVM编译：

  • 举一个java中令人印象深刻的特性：

    1、分层编译

    • 在Java 7以前，需要研发人员根据服务的性质去选择编译器。对于需要快速启动的，或者一些不会长期运行的服务，可以采用编译效率较高的C1，对应参数-client。

      长期运行的服务，或者对峰值性能有要求的后台服务，可以采用峰值性能更好的C2，对应参数-server。

    • Java 7开始引入了分层编译的概念，它结合了C1和C2的优势，追求启动速度和峰值性能的一个平衡。分层编译将JVM的执行状态分为了五个层次。五个层级分别是：

      • 解释执行：在这个层次，JVM 会直接解释 Java 字节码并执行代码。
      • 执行不带 profiling 的 C1 代码：在这个层次，JVM 会使用 C1 编译器将代码编译成本地代码，并执行该代码。这个层次的代码通常速度较快，但精度较低。
      • 执行仅带方法调用次数以及循环回边执行次数 profiling 的 C1 代码：在这个层次，JVM 会在编译代码时添加一些 profiling 信息，如方法调用次数和循环回边执行次数。这些信息可以用来优化代码并提高执行速度。
      • 执行带所有 profiling 的 C1 代码：在这个层次，JVM 会使用更多的 profiling 信息来编译代码，并进一步优化代码。
      • 执行 C2 代码：在这个层次，JVM 会使用 C2 编译器将代码编译成高度优化的本地代码，并执行该代码。这个层次的代码通常速度最快，但编译时间较长。
        • 图中第①条路径，代表编译的一般情况，热点方法从解释执行到被3层的C1编译，最后被4层的C2编译。
        • 如果方法比较小（比如Java服务中常见的getter/setter方法），3层的profiling没有收集到有价值的数据，JVM就会断定该方法对于C1代码和C2代码的执行效率相同，就会执行图中第②条路径。在这种情况下，JVM会在3层编译之后，放弃进入C2编译，直接选择用1层的C1编译运行。
        • 在C1忙碌的情况下，执行图中第③条路径，在解释执行过程中对程序进行profiling ，根据信息直接由第4层的C2编译。
        • 前文提到C1中的执行效率是1层>2层>3层，第3层一般要比第2层慢35%以上，所以在C2忙碌的情况下，执行图中第④条路径。这时方法会被2层的C1编译，然后再被3层的C1编译，以减少方法在3层的执行时间。
        • 如果编译器做了一些比较激进的优化，比如分支预测，在实际运行时发现预测出错，这时就会进行反优化，重新进入解释执行，图中第⑤条执行路径代表的就是反优化。

    2、编译优化

    • （SSA IR）Java编译中的中间表达形式：静态单赋值（Static Single Assignment，SSA）IR，SSA 中间表达形式是一种图结构IR，特点是每个变量只能被赋值一次，可以识别出冗余变量和死代码。

      我们可以将编译器的每一种优化看成一个图优化算法，它接收一个IR图，并输出经过转换后的IR图。编译器优化的过程就是一个个图节点的优化串联起来的。

程序计数器：

  • 本质：存储区域是一块很小的内存空间，存储的内容其实就是下一个指令的地址，线程之间不共享。
  • 作用：
    • 字节码解释器改变这一区域的值，实现代码流程控制。
    • 多线程下存储当前线程运行位置，等线程切回来时能够继续运行。
  • 注：程序计数器是唯一一个不会出现 OutOfMemoryError 的内存区域，因为本质就是存储指令地址的空间。没有对内存空间扩增的需求

    • CAS配上失败重试

    • 本地线程分配缓冲：每个线程分配一个TABLE，只有TABLE用完才会同步锁定。

    • 逃逸分析：
      • 生命周期：分配在堆上的的生命周期是整个程序，而分配在栈上的生命周期是一个方法的执行
      • 解决问题：解决有些对象的生命周期仅限于某个方法的情况。这样的对象如果被放在堆上会造成一些浪费
      • 解决方法：用于判断某些对象需要在栈上分配还是在堆上分配
    • 垃圾回收：
      • 引用计数算法：当一个对象被创建或被引用时，计数+1，当不再被引用时，计数-1。
      • 根搜索法：从根对象开始遍历所有存活对象，当对象不再被引用时，回收。

索引失效的几种原因：

总结不生效有两种情况：

1、索引本质就是按照某个字段为数据排序，如果搜索条件用不到拍好的序，就不走索引。

2、当扫描了数据10%～30%左右的时候，优化器就会放弃索引，自动全表扫。

参考：https://www.51cto.com/article/702691.html

1、联合索引不满足最左匹配原则：

  • 索引：KEY union_idx(id_no,username,age)

  • 失效语句：explain select * from t_user where username = 'Tom2' and age = 12;

  • 根本原因：当创建（A,B,C)索引的时候，数据库首先会根据A排序，然后再根据B和C排序。所以如果查询的时候不查A字段的话，就无法匹配索引，导致索引失效。

    另外当创建(A,B,C)索引的时候，其实只有查(A,B,C)的时候才是最快的。

2、通过SELECT索引特定字段可以实现走覆盖索引，如果没有特定字段就会失效：
  • 索引：KEY union_idx(id_no,username,age)
  • 走索引语句：explain select id_no, username, age from t_user where username = 'Tom2';
  • 失效语句：explain select * from t_user where username = 'Tom2';
  • 根本原因：创建索引的时候，其实就是将id_no, username, age这三列的数据，根据规则放到特定的数据结构中，所以当那这些数据的时候，查询优化器可以直接从索引中拿到对应的值，减少了回表的时长。
3、索引列使用函数计算：

  • 索引：KEY union_idx(id)

  • 失效语句：explain select * from t_user where id + 1 = 2 ;

  • 根本原因：

    • 索引列使用了函数之后，由于数值改变，需要原先索引的顺序会发生变化，所以会全表扫。

    • 非确定性函数（如 RANDOM() 或 NOW()），会在每次调用时产生不同的结果。由于数据库优化器无法预测这些结果，它可能不会选择使用索引。

4、LIKE错误使用：
  • 失效语句：like '%abc';ike '%abc%';
  • 根本原因:
    • 索引创建时是按照从左往右创建的，由于在左侧使用了占位符，原先的排序就不生效了。
5、OR不正当使用：
  • 失效语句1：explain select * from t_user where id = 2 or username = 'Tom2';
  • 失效语句2: explain select * from t_user where id > 1 or id < 80;
  • 根本原因：
    • 失效语句1: username没有索引，所以会全表扫
    • 失效语句2: 当使用 OR 条件连接多个范围查询时，如果两个范围之间有很多数据重叠，查询涵盖的数据量会变大，使得索引的选择性降低。此时，数据库优化器可能会认为全表扫描比使用索引更高效。
6、两列比较：
  • 失效语句：explain select * from t_user where id > age;
7、不等于和is not null，not exist：

  • 失效语句1：explain select * from t_user where create_time != '2022-02-27 09:56:42';

  • 失效语句2：explain select * from t_user where id_no is not null;

  • 根本原因：

    • 当作为字符串的时候，用 != 可能会不走索引：

      1. 索引选择性不佳：当使用 !=（或 <>）操作符时，查询涵盖的数据范围可能相当大，这导致索引的选择性降低。数据库优化器可能认为全表扫描比使用索引更高效。如果查询涉及的数据范围较小，那么索引仍有可能被使用。
      2. 索引顺序与查询顺序不匹配：当 !=（或 <>）操作符与其他条件（如排序和分组）结合使用时，索引失效的可能性更高。因为这些操作要求数据按照特定顺序处理，而索引的顺序可能与这些操作所需的顺序不匹配。

    • Is not null 不走索引：

      1、索引选择性不佳：当大部分行的列值都不为 NULL 时，使用 IS NOT NULL 作为过滤条件可能覆盖了大量数据。在这种情况下，索引的选择性降低，数据库优化器可能会认为全表扫描比使用索引更高效。

      2、数据库优化器决策：根据统计信息和成本评估，数据库优化器可能会判断全表扫描或其他扫描方法（如顺序扫描）比使用索引更合适。优化器会根据数据分布、表大小、硬件资源等因素做出决策，这可能导致 IS NOT NULL 不走索引。

      3、索引类型：某些索引类型可能无法用于 IS NOT NULL 查询。例如，MySQL 的 MyISAM 存储引擎不会为全为 NULL 的列创建索引记录。因此，在使用 MyISAM 存储引擎的表中，IS NOT NULL 查询可能无法使用索引。

1、项目盘点—困难、亮点

一次优化经历：

  • Mybatis批量更新手动管理事务

    由于之前是通过@Transactional 将大方法包了起来实现的事务回滚，但如果方法执行时间很长，事务将会保持打开状态，占用数据库连接资源。所以为了解决这个问题同时又要能够实现事务回滚。对其中的批量更新操作优化。优化方法是使用Mybatis批处理方法。这样在一定程度上提升了更新速度但是当更新的数据量很大的时候任然会有一定的风险。因为在commit之前，Mybatis都会讲数据库的操作保存在内存的队列中，当执行commit之后才会将操作发给数据库。当数据量很大的时候，一是会占用大量的内存资源，二是数据库执行的时候可能会因为执行时间过长导致锁冲突，三是当遇到异常需要回滚的时候，一次性回滚的数据量可能是巨大的，这也会增加成本。解决方法就是在for循环中分批提交。相比于一次性插入，分批提交能够减少内存消耗，减轻数据库的负担同时减少回滚的成本。

  • 使用內联视图对表关联更新的优化

    情景是有两个表一个是客户表，一个是订单表。要通过订单表关联客户表，更新订单表字段。之前的sql是通过子查询来实现的。但由于数据量变多，子查询方式实现的成本也就提高了，因为每次都要去和子查询的结果做匹配才能够找到要更新的字段，其中也包含了很多重复计算。通过使用oracle的内联视图，可以提升关联表更新的性能，因为 Oracle 的查询优化器可以更好地理解内联视图，并对其进行更有效的查询计划生成。通过内联视图，Oracle 可以在执行计划中进行更好的优化和处理，可能会利用更多的索引和优化技术，获得更高效的查询性能。

  • 业务逻辑整合

    在一个逻辑中，类似于分配任务，任务是有编号的。但每个人分配到的只是数量，需要后台将对应数量的任务分配上去。原先逻辑是先将符合条件的任务查出来，最后再做更新。我看逻辑其实可以整合，最后就整合成一个更新sql，每次吧符合条件的前几条数据更新上去。

内部方法：

  • 如何确定哈希桶数组索引位置：取key的hashCode值、高位运算、取模运算。

    • 根本问题：最大限度地减少哈希冲突，提高查找和存储元素的效率。

    • 减少哈希冲突：

      高位运算：取模一般都是取的低位的值。通过高位^低位的方式，可以将高位的影响向低位扩散，从而使得地位更具有随机性。

    • 提高查找和存储元素的效率：

      取模的算法：

      当length总是2的n次方时，h& (length-1)运算等价于对length取模，也就是h%length，但是&比%具有更高的效率

JDK1.8 Resize扩容机制优化

  • 初始化一个新的Entry数组，将老的数据中的值转移到这个新的Entry中
  • JDK1.7：对每一个key重新hash一遍，确定在新的Entry中的索引位置
  • JDK1.8：使用二次幂扩展的时候，如果一个key的hash新增计算的那个bit位是0，则索引位置与老的Entry保持不变，如果新增计算的bit位是1，则索引位置index=oldIndex+oldCup

小结：

  1. 扩容是一个特别耗性能的操作，所以当程序员在使用HashMap的时候，估算map的大小，初始化的时候给一个大致的数值，避免map进行频繁的扩容。
  2. 负载因子是可以修改的，也可以大于1，但是建议不要轻易修改，除非情况非常特殊。
  3. HashMap是线程不安全的，不要在并发的环境中同时操作HashMap，建议使用ConcurrentHashMap。
  4. JDK1.8引入红黑树大程度优化了HashMap的性能。
  5. 还没升级JDK1.8的，现在开始升级吧。HashMap的性能提升仅仅是JDK1.8的冰山一角。

框架的SpringBoot启动流程、循环依赖

本质： Web服务+Sevlet容器

  • 本质目的：访问远程主机上的资源
  • 实现方式：通过Web服务实现将主机上的资源通过url暴露、通过Sevlet容器实现对访问请求的处理（也就是后端程序）

tomcat本质是用来处理请求的。

SpringBoot最重要的一件事情就是实现了嵌入式Tomcat。

1、Tomcat独立启动一个Spring的服务方式。

  • 首先会通过配置ContextLoaderListener启动SpringIOC容器。

    • 如果在配置文件web.xml中配置了ContextLoaderListener作为监听器之后，因为ContextLoaderListener实现ServletContextListener接口（也可以说是ContextLoaderListener遵循了Tomcat规范），所以Tomcat在启动的时候会通知ContextLoaderListener，并且调用contextInitialized()方法。
    • contextInitialized()方法会根据配置文件中的WebApplicationContext值来创建对应的Web容器，例如Spring的XmlWebApplicationContext
    • 接着就会执行spring的著名的初始化方法refresh()：
      1. 加载Bean定义：refresh()方法会读取配置文件或注解，解析并加载Bean的定义。它会扫描所配置的Bean定义，创建对应的BeanDefinition对象，并注册到IOC容器中。
      2. 创建和初始化Bean实例：refresh()方法会根据Bean的定义，实例化和初始化相应的Bean对象。它会根据依赖关系进行对象的创建和组装，执行各个Bean的初始化回调方法（如InitializingBean接口或自定义的初始化方法）。
      3. 解析和应用Bean之间的依赖关系：refresh()方法会解析Bean之间的依赖关系，确保依赖的Bean对象已经被实例化并设置到相应的属性中。
      4. 执行其他的配置和处理逻辑：refresh()方法还会执行其他的配置和处理逻辑，例如注册BeanPostProcessor（后置处理器）、解析AOP配置、处理事件监听器等。

  • 通过DispatcherServlet启动了SpringMVC容器，并将SpringIOC容器设置为父容器。

    • 执行父类HttpServletBean的init()方法
    • 执行子类FrameworkServlet类的initServletBean()方法
    • 执行DispatcherServlet类的onRefresh()方法，初始化springmvc的配置

2、SpringBoot内置Tomcat之后的启动方式

  • 创建SpringApplication对象，读取并加载应用程序的配置信息：
    • 创建一个新的SpringApplication对象。
    • SpringApplication会根据主类上的注解和配置，扫描应用程序的类路径，加载配置文件（如application.properties或application.yml），并解析应用程序的配置信息。
  • 创建并配置Spring应用程序上下文对象，加载和初始化应用程序的Bean，执行相关的生命周期方法：
    • 根据应用程序的配置信息，创建一个合适的ApplicationContext对象，通常是AnnotationConfigApplicationContext或其他派生类。
    • ApplicationContext会根据配置信息和类路径上的内容，扫描并加载应用程序的Bean定义，包括通过注解配置的Bean、XML配置文件中的Bean等。
    • ApplicationContext会根据Bean定义实例化Bean，并进行依赖注入、初始化和销毁等相关操作。
    • 执行相关的生命周期方法，如@PostConstruct和@PreDestroy注解标记的方法。
  • 创建内嵌的Tomcat服务器对象，并设置服务器相关的属性：
    • 创建一个Tomcat对象，作为内嵌的Tomcat服务器。
    • 配置Tomcat的属性，如端口号、上下文路径、连接超时等。
  • 将Spring应用程序上下文对象与Tomcat服务器进行关联，设置Tomcat的上下文路径：
    • 将ApplicationContext设置到Tomcat中，使得Tomcat可以访问应用程序的Bean和资源。
    • 设置Tomcat的上下文路径，即应用程序在Tomcat中的访问路径。
  • 配置Tomcat服务器的路由和处理器，将HTTP请求分发给相应的处理器：
    • 配置Tomcat的路由和处理器，将不同的HTTP请求映射到对应的处理器（Servlet或Spring MVC控制器）。
    • 配置Tomcat的路由规则，使得请求能够被正确地分发到相应的处理器进行处理。
  • 启动Tomcat服务器，监听指定的端口，等待接收和处理HTTP请求：
    • 调用Tomcat的start()方法启动服务器，开始监听指定的端口，等待接收和处理HTTP请求。
    • 当Tomcat服务器接收到HTTP请求时，根据路由规则将请求分发给相应的处理器进行处理，然后生成响应返回给客户端。

要理清楚SpringBootSpringMVC和启动流程中Tomcat不同的启动方式，首先要弄清楚Tomca的作用，其实就是提供了一种访问远程主机资源的方式，本质组成就是Web服务+Servlet容器，其中Web服务是可以将主机上的资源通过URL路径暴露出来，供客户端通过HTTP协议进行访问。再通过Servlet容器实现对访问请求的处理。然后对于SpringMVC的启动方式，在Tomcat启动的时候其实启动了两个容器，一个是SpringIOC容器，另一个就是SPringMVC容器。在Tomcat启动的时候，会根据.xml配置文件中配置的监听器，调用对应的初始化方法，并根据配置来创建对应的Web容器。创建好Web容器后便开始了Spring的初始化。与Spring容器类似的，通过一系列的调用最后通过DispatcherServlet启动了SpringMVC容器，并将SpringIOC设置为他的父容器。

后来在SpringBoot中，通过实现了内嵌式的Tomcat，实现了jar包直接部署。具体的Tomcat启动流程：首先会创建SpringApplication对象，读取相关的配置文件，接着创建Spring上下文对象，执行Bean的初始化工作。

这里将SpringMVC作为SpringIOC的子容器有三个好处，首先是松耦合和模块化，因为SpingMVC只关注处理Web请求和生成响应，这种模块化的设计使得代码更加清晰、可维护性更高。其次是Spring MVC作为Spring IOC的子容器，可以共享Spring IOC容器中的资源和上下文。再次是SpringMVC可以依赖SpingIOC实现统一生命周期管理。当程序在启动过程中具备了正确的配置和必要的资源之后，就为Tomcat服务创建提供了基础。通过创建Tomcat对象，设置端口号等相关配置，然后将上下文对象设置到Tomcat中，使得Tomcat可以访问应用程序的Bean和资源，同时设置程序中Tomcat的访问路径。接着配置Tomcat服务的路由，启动服务器开始监听指定的端口。
